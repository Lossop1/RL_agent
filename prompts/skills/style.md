# 代码风格

## 通用规则

1. **遵循现有代码风格** — 匹配项目中已有文件的风格。不要引入新的模式或格式。

2. **禁止 emoji** — 以下地方不得出现 emoji：
   - 代码注释
   - 打印/输出字符串
   - 日志消息
   - 错误消息
   - 文档字符串
   - 任何用户可见的文本

3. **保持一致** — 不确定时，复制同目录下最近文件的风格。

# 输出格式约
使用以下标记来组织面向用户的输出。main.py 识别这些标记并渲染。

| 标记 | 用途 | 使用场景 |
|------|------|---------|
| `[INFO]` | 一般信息 | 配置摘要、模板列表、操作结果、训练已启动 |
| `[ERROR]` | 错误和警告 | 训练失败、NaN检测、进程丢失、环境检查失败 |
| `[PROGRESS]` | 进度更新 | 当前轮数、下次分析预计时间，一行 |
| `[DIAGNOSIS]` | 结构化诊断报告 | 完整诊断结论：根因、证据、置信度、建议 |
| `[ASK]` | 询问用户决策 | 是否从检查点恢复、确认调参方案、确认模板使用 |
| `[DONE]` | 任务完成摘要 | 任务完成时的最终摘要 |

规则：
- 每次输出只用一种标记。不要混用。
- `[ASK]` 必须是是非题或单选。用户下一轮输入。
- `[PROGRESS]` 一行：当前轮数、目标轮数、ETA。
- `[DIAGNOSIS]` 可多行，分段结构。
- `[ERROR]` 必须包含原因，要可操作。
